## Methods

### Artifact

We deployed *Mayor Climate Game* as a browser-based, single-page application hosted on Vercel at `https://mayor-climate-game.vercel.app`. The artifact required no installation; participants opened the public URL in a standard web browser.

Participants acted as a city mayor with a fixed budget of **CNY 10 million** (remaining budget displayed in the left panel). The city was represented as a **12 × 12** grid with five terrain types visible on the map: normal zones, residential areas, ventilation corridors, wind outlets, and low basins (color-coded in the interface). Participants selected one of three facility types from the left panel—**coal power** (80 万元, 120 t CO₂e), **wind power** (200 万元, 0 t CO₂e), or **ground-source heat pumps** (160 万元, 15 t CO₂e)—and clicked grid cells to place facilities; clicking an occupied cell removed the facility and refunded its cost.

During gameplay, the interface displayed:

- **Remaining budget** (left panel)
- **Three real-time city-level indicators** (right panel): average temperature (°C), average wind speed (m/s), and total carbon emissions (t), each with a categorical status label (e.g., stable / overheating, good / blocked, low / high)
- A **step-level impact card** (“本步影响”) that updated after each placement or removal, reporting whether the latest action was overall favorable, unfavorable, or neutral with respect to temperature, wind, carbon, and budget changes
- A **rule-based advisor panel** labeled **“规则型提示”** (not a live LLM during gameplay), which showed up to four scripted tips triggered by layout and metric thresholds (e.g., coal at a wind outlet, high carbon, low wind speed)
- A **context tip line** and **overall advice line** below the map area, also generated by fixed if–then rules over the current layout and metrics

When participants clicked **Finish** (结算), a modal dialog opened with a **two-item post-play survey** (see Design). Before download, the interface also showed the session score in the main panel. Participants could then:

- download a human-readable **TXT report** (layout, metrics, score, survey answers, and any optional AI summary text), and/or
- download a machine-readable **JSON file** containing the full session record.

On TXT or JSON download, the client attempted to POST the session payload to a Vercel serverless endpoint (`/api/submit`), which inserted a row into a Supabase table (`public.game_results`) when backend credentials were configured. Stored fields included `project`, `budget_used`, `score`, `metrics` (JSON), `layout` (JSON array of `{x, y, terrain, facility}`), `survey` (JSON), and `raw_result` (full client payload). If Supabase submission failed, local download still proceeded.

**Material boundaries (simulation and “AI” components).** The deployed MVP used a **simplified, rule-based microclimate model** implemented in client-side JavaScript. It was **not** a CFD, WRF, GIS, or other professional urban-climate simulation; temperature, wind, and carbon values were heuristic functions of facility parameters, terrain modifiers, and placement locations. All in-game feedback during placement—impact cards, advisor tips, context tips, and overall advice—was **curated, deterministic scripting**, not model inference from an external AI service.

Separately, the post-play dialog offered an optional button **“生成 AI 总结.”** When clicked (and when `DEEPSEEK_API_KEY` was configured on the server), the client called `/api/chat`, a Vercel proxy that sent a short summary of the session (score, budget used, metrics, facility count, and the two survey answers) to the **DeepSeek Chat API** (`deepseek-chat`, temperature 0.4, max 360 tokens) and displayed the returned text. This step was **optional**, occurred **after** gameplay, and was **not** used to drive in-game mechanics. If the API was unavailable, the interface showed an error message and gameplay data remained downloadable without an AI summary.

On first visit, participants saw a **30-second onboarding modal** (game instructions in four steps) unless they had previously closed it (`localStorage` key `mayorClimateGuideSeen`). The modal included **Skip for now** and **Start Game**; it did **not** by itself constitute a research consent form in the deployed build (see Consent).

### Design

The deployed artifact supported a **single-session gameplay study** with **post-play self-report** and **automatic behavioral logging**. The publicly deployed version did **not** include a built-in pre-test instrument or a separate control condition.

**Exposure (implemented in the artifact).** All participants used the same instrument: one round of *Mayor Climate Game* with real-time metric updates, step-level impact feedback, and rule-based advisory messages during facility placement.

**Post-play survey (embedded in the web tool, administered after Finish).** Two required closed-ended items were presented in Chinese:

1. *Did you choose coal power in order to save budget?* (`budgetChoice`: yes / no)
2. *Placing coal power at a wind outlet reduces city-wide ventilation.* (`windUnderstanding`: correct / incorrect)

There were **no open-ended survey fields** in the deployed version.

**Behavioral and system-recorded variables** (computed or stored automatically per session):

| Variable | Source / definition in deployed build |
|----------|----------------------------------------|
| `budgetUsed` | 1000 − remaining budget (万元) |
| `score` | Composite score from carbon, temperature deviation from 24 °C, and wind speed (client-side formula in `getScore()`) |
| `metrics.temp`, `metrics.wind`, `metrics.carbon` | Final city-level averages after last placement |
| `layout` | List of placed facilities with grid coordinates and terrain type |
| Facility counts / terrain-specific placements | Derivable from `layout` (e.g., coal on `outlet` or `corridor`) |
| `survey.budgetChoice`, `survey.windUnderstanding` | Post-play modal responses |
| `aiSummary` | Text returned by DeepSeek if the participant clicked **生成 AI 总结**; otherwise empty |
| `timestamp` | ISO timestamp in exported JSON |

**Pre/post structure in the deployed build.** The live tool implemented **post-play measurement only** (two survey items plus behavioral logs). It did **not** administer a pre-test before gameplay.
